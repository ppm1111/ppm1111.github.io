變數命名

對於電腦來說，變數名稱並不重要，變數是給人看的，變數的命名好壞將影響整個程式的可讀性及可維護性．

Indeed, the ratio of time spent reading versus writing is well over 10 to 1. We are constantly reading old code as part of the effort to write new code. …[Therefore,] making it easy to read makes it easier to write. - Robert C. Martin

上面的意思是讀寫比例超過10:1，擁有好的可讀性將會使程式開發效率大幅上升。

變數命名的好壞可由以下2點判斷:

  • 可讀性
  • 一致性

可讀性

變數是影響可讀性最大的一環，例如計算圓形面積:

// 將秒數轉換成時:分:秒 ex 輸入10000轉成02:46:40
function a(b) {
    let c = parseInt(b / 3600);
    let d = parseInt(b % 3600 / 60);
    let e = parseInt(b % 3600 % 60);
	return `${c}:${d}:${e}`;
}

上述的程式碼在無註解的情況不容易理解有什麼功能，若是面對更複雜的function或是整個程式，根本無法讀懂程式的運作。

好的命名:

function convertSecondsToHMS(totalSeconds) {
    let hours = parseInt(totalSeconds / 3600);
    let minutes = parseInt(totalSeconds % 3600 / 60);
    let seconds = parseInt(totalSeconds % 3600 % 60);
	return `${hours}:${minutes}:${seconds}`;
}

一致性

一致性是指必須要用相同的規則命名整個程式的變數，可以是合作的工程師口頭約定也可以是一份code style文件．

如果採用了駝峰式就必須一直使用駝峰式，如果將使用者名稱命名成userName就不要在中途改成accountName或只剩下user．

變數命名規則

變數的字元是可以包含任何合法的標識符-無限長度的Unicode字母，例如一些拉丁字母或是中文:

let ಠ_ಠ = 1;
let 測試 = 1;
let $test = 1;
let _ts12et$ = 1

但一般情況下應以下劃線_及大小寫英文、數字為主，不能以數字開頭。

let user_name = 'John';
let userName = 'John';

主流命名規則

駝峰式命名法(Camel Case)

第一個字母為小寫，之後每一個單字的開頭為大寫，不包含空格。

和Snake Case是較多被採用的命名方式，擇一使用即可。

例如:

  • userName
  • fileName

Pascal命名法(Pascal Case)

跟Camel Case類似，只是第一個字要大寫。

例如:

  • UserName
  • FileName

Snake Case

單字皆為小寫，單字以底線_分離。

和Camel Case是較多被採用的命名方式，擇一使用即可。

例如:

  • user_name
  • file_name

Kebab Case

跟Snake Case類似，單字以減號-分離。

常用地方:

  • html的裡面的class和id

    <div class="col-md-6" id="user-password"></div>

  • restful api的網址分隔符號，如: api/order-list。

例如:

  • user-name
  • file-name

Screaming Case

跟Snake Case類似，單字皆為全大寫。

常用地方:

  • 常量變數(constant variable)，如: MAX_EXECUTION_TIME=60

  • 枚舉名稱(enum)

    enum Level {
  LOW,
  MEDIUM,
  HIGH
}

例如:

  • USER_NAME
  • FILE_NAME

正確使用命名

好的命名並不容易，當場景越來越複雜時，就會越難使用幾個單字描述該變數的功能。

以下是可以遵守的規則:

  • 不要使用長度超過50字元的變數，太長的變數反而使程式碼可讀性降低，在清楚表達的前提下，名稱越短越好。
  • 變數的組成應為多個形容詞、名詞組成，避免使用單一動詞當作變數，如create = 1。
  • 區分單複數，例如: user是單數，users、userArray、userList、userCollection是複數。
  • 不要加入無意義的前綴，如: m_name，這寫法通常代表一個class下的成員變數(member variable) name，但是不必要，一個class底下的name只可能是成員變數，使用name即可。
  • 盡量不要使用無意義的單字，data、temp、value等，這種類型的單字稱作干擾詞，若是遍及整份程式碼，掩蓋了具有意義的單字。
  • 使用可以發音的單字，在溝通時若選用難以發音的單字，就會增加溝通的難度。
  • 若要使用無意義的單字或是單個字母如: i, j等，要確保變數的作用域在短行數的程式碼內，如for loop或一個短的if block，當變數的作用域越廣則須越長的名稱清楚地描述該變數的意義，反之，越狹窄的話名稱則要越短。
  • 推薦使用Camel Case或是Snake Case命名方式
  • 使用英文單字而不是類似中文拼音的方式，如biaoGe(中文拼音 的表格) 。
  • 使用有意義或公認的縮寫而不是只有少部分人才懂得縮寫，如果不是就應該使用全寫。公認的縮寫像是id(identity)、int(integer)這類的，不好的縮好像是order system裡面的user寫成os_user或soUser，或是db這類的縮寫，比較好的是system_order_user或systemOrderUser。
  • 當進行開發時，如果有更適合的變數名稱，就去更改他，不要害怕隨時改名。
  • 選擇一個單字就堅持下去，例如: user跟member、get跟retrieve跟fetch。
  • 使用正確的資訊專業術語，如某種演算法或是Factory、Interface，會去看程式碼的人都是有專業背景的，使用專業術語更能清楚表達，反之，若使用錯誤將產生極大的混淆。
  • 在命名時，應注重可搜索性，開發上難以避免搜尋變數，若是將大量的變數命名成data，即使不衝突也會造成搜尋上的困擾。
  • 標示變數的相關訊息可幫助了解變數的狀態，例如: boolean的變數取名為isLogin，是object的變數取名為userObj，是array的變數取名為userList或是userArray，比起users要清楚一點，相關訊息並不是指型態，所以不會出現userString或是strUser等(類似匈牙利命名法 Hungarian Notation)。
  • boolean的變數可使用is/has/can + 名詞或形容詞或enable/disable + 動詞，以表達行為，如：isUser、hasName、canSignup、hidden、available、enableOrder、disableUpate，若是function返回boolean值可在前面加check，如:checkHasName()、checkIsUser()。
  • 變數命名方式因語言而異，用甚麼語言就用那語言推薦的style。
